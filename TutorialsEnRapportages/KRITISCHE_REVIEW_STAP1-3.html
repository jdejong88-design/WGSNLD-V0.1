🔧 Verbetering 2: Tokenizer Validatie

Probleem: We hebben de tokenizer gebouwd en toegepast, maar valideren niet: - Hoeveel tokens per artikel gemiddeld (essentieel voor sequence length planning) - Token frequency verdeling (zijn rare tokens goed?) - Out-of-vocabulary rate (hoe vaak ontbreken tokens?)

Voorstel: Voeg validatiestappen toe in 02_tokenization.py (of nieuw script):

# Voeg toe aan tokenizer script: mean_tokens_per_article = total_tokens / num_articles max_tokens = max(token_counts) print(f"Avg tokens/article: {mean_tokens_per_article:.0f}") print(f"Max tokens: {max_tokens}") print(f"OOV rate: {oov_count / total_tokens * 100:.2f}%")

Waarom beter: Dit geeft je direct inzicht dat de tokenizer correct werkt. Als avg tokens/artikel > 512, weet je dat je sequence length moet verhogen. Dit voorkomt stilzwijgend gegevensverlies.

⚠️ Kritieke Bevindingen

Issue 1: Geen Error Handling in Forward Pass

Probleem: In 03_build_transformer_model.py forward() functie: - Geen check of input_ids > max_seq_length - Geen validatie van input shape - Geen warning als attention weights explodeerden

Voorstel: Voeg shape validatie toe aan forward():

def forward(self, input_ids, attention_mask=None): batch_size, seq_length = input_ids.shape # VALIDATIE TOEVOEGEN: assert seq_length <= self.max_seq_length, \ f"Seq length {seq_length} > max {self.max_seq_length}" assert input_ids.dtype == torch.long, \ f"Input must be torch.long, got {input_ids.dtype}" # Rest van forward...

Waarom beter: In training kan een dataloader bug subtiel foute shapes doorsturen. Early assertions voorkomen sloppy errors die 2 uur later opduiken.

❓ TOESTEMMING NODIG: Zal ik shape validatie toevoegen aan forward()?

Mogelijke Verbeteringen

🔧 Verbetering 3: Gradient Flow Monitoring

Probleem: Geen code die monitort of gradiënten goed door het netwerk vloeien. Met 8 lagen en attention kan je: - Vanishing gradients krijgen (dieper in netwerk = kleinere gradients) - Exploding gradients (vermenigvuldiging van gewichten)

Voorstel: Voeg gradient norm logging toe in training script (04_train_model.py):

# In training loop: loss.backward() total_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0) if batch_idx % 100 == 0: print(f"Grad norm: {total_norm:.4f}")

Waarom beter: Gradient norms vertellen je of het model stabiel leert. Norms > 10 betekent mogelijk exploding gradients, norms < 0.0001 betekent vanishing gradients.

❓ TOESTEMMING NODIG: Zal ik gradient norm logging toevoegen aan training?
