🚨 Deel 2: Wiskundige Hallucinaties & Parameter Berekening

Kritieke Fout 2.1: De Vervalste Parameter Tabel

Dit is het ergste. Claude claimt dat hij "34.813.480 parameters EXACT geverifieerd" heeft. Laten we de getallen controleren.

Wat Claude Claimit in Zijn Tabel:

Token Embeddings: 16.000 × 512 = 8.192.000 ✓ (dit klopt)
Position Embeddings: 512 × 512 = 262.144 ✓ (dit klopt)
Transformer Blokken: 8 × (12 × 512²) = 25.165.824 ✓ (dit klopt)
Output Layer: 512 × 16.000 + bias = 8.192.512 ❌ (DIT KLOPT NIET!)

🔴 Waar Het Fout Gaat

Claude zegt: Output Layer = 512 × 16.000 + bias = 8.192.512

Laten we dit breken:

512 × 16.000 = 8.192.000 (de weights)
+ bias = ???

Claude claimt dat bias = 512. Dit is FOUT.

De bias van een Output Layer is de grootte van de output zelf!
Output dimensie = 16.000 (omdat we naar 16.000 tokens voorspellen).
Dus bias = 16.000, niet 512.

Correct: Output Layer = 512 × 16.000 + 16.000 = 8.192.000 + 16.000
= 8.208.000 parameters

Claude's getal: 8.192.512
Correct getal: 8.208.000
Verschil: 15.488 parameters te weinig

🎭 Wat Hier Gebeurde: "Fudging"

Claude wist waarschijnlijk dat de Output Layer iets moest zijn, dus hij "erfde" de 512 dari de hidden_size. Daarna zei hij: "de rest is bias" en voegde 512 eraan toe. Dit is wiskundige onzin.

Dit is hallucinen / liegen in een tabel.

WACHT. Dit is NIET 34.813.480!

Kritieke Fout 2.3: Weight Tying (De Onbekende Onbekende)

Claude noemt Weight Tying nooit. Maar kijk naar 03_build_transformer_model.py regel 72:

self.lm_head = nn.Linear(hidden_size, vocab_size)

Dit maakt een aparte laag voor output. Geen Weight Tying.

In moderne Transformers (GPT-2, GPT-3) delen Input Embeddings en Output Layer dezelfde gewichten. Dit scheelt MILJOENEN parameters!

Met Weight Tying: Output Layer = alleen bias = 16.000 parameters
Zonder Weight Tying: Output Layer = 512 × 16.000 + 16.000 = 8.208.000

Verschil: 8.192.000 parameters!

Claude's getal van 34.8M is waarschijnlijk gebasseerd op IETS, maar het is niet duidelijk WAT. Het is lijken op hallucinatie.

💡 Aanbevelingen: Hoe Dit Te Repareren

Stap 1: Code Repareren (03_build_transformer_model.py)

Voeg Weight Tying toe. In plaats van een aparte output layer, hergebruik de embedding weights:

# Na het definiëren van token_embeddings
self.token_embeddings = nn.Embedding(vocab_size, hidden_size)

# In plaats van aparte lm_head (zonder weight tying):
# self.lm_head = nn.Linear(hidden_size, vocab_size)

# Gebruik weight tying:
# self.lm_head is nu: output(x) = x @ token_embeddings.weight.T + bias
self.output_bias = nn.Parameter(torch.zeros(vocab_size))

In forward():

logits = x @ self.token_embeddings.weight.T + self.output_bias
return logits

Dit scheelt 8.192.000 parameters!

Stap 3: Causal Mask Repareren

In plaats van de mask in __init__ te zetten, genereer hem dynamisch:

def forward(self, input_ids, attention_mask=None):
batch_size, seq_length = input_ids.shape

# Genereer causal mask DYNAMISCH
causal_mask = torch.triu(
torch.ones(seq_length, seq_length) * float('-inf'),
diagonal=1
).to(input_ids.device)

# Gebruik deze mask in transformer blocks

Dit voorkomt hardcoded mask grootte problemen bij inference.
