🔧 Wijziging 1: Weight Tying Implementatie (Modern Best Practice)

📍 KRITIEKE WIJZIGING: Output Layer Herstructurering

Wat Was Het Probleem?

Original code (FOUT): De output layer was een aparte nn.Linear(hidden_size, vocab_size) laag.

Dit betekent: 512 × 16.000 + 16.000 = 8.208.000 parameters alleen voor output!

Dit is ~24% van alle parameters, terwijl het alleen maar gewichten kopieert van Token Embeddings.

De Oplossing: Weight Tying

Concept: Input Embeddings en Output Layer delen dezelfde gewichten.

Modern practice: GPT-2, GPT-3, BERT — allemaal gebruiken Weight Tying.

Voordeel: Bespaart 8.2 miljoen parameters zonder enig verlies van capabiliteit.

Code-wijziging: Waar Precies?

📁 File: 03_build_transformer_model.py
📍 Locatie: Regels 70-72 (in __init__ methode)
❌ VOOR (Fout)
# Output layer self.layer_norm = nn.LayerNorm(hidden_size) self.lm_head = nn.Linear(hidden_size, vocab_size)
✅ NA (Correct)
# Output layer with weight tying # (share weights with token embeddings) self.layer_norm = nn.LayerNorm(hidden_size) self.output_bias = nn.Parameter( torch.zeros(vocab_size) )

Waarom Dit Werkt?

In plaats van een matrix van 512 × 16.000 gewichten + 16.000 bias, gebruiken we nu alleen de bias (16.000 parameters).

De eigenlijke gewichten komen van token_embeddings.weight in de forward pass.

Resultaat

Output parameters: 8.208.000 → 16.000

Besparing: 8.192.000 parameters (97% reductie!)

🔧 Wijziging 2: Weight Tying in Forward Pass

📍 KRITIEKE WIJZIGING: Logits Berekening

Wat Was Het Probleem?

Original code gebruikte een aparte laag voor output:

logits = self.lm_head(x)

Dit doet: x @ W + b, waar W 8.2M parameters heeft.

De Oplossing

Gebruik de embedding matrix rechtstreeks:

📁 File: 03_build_transformer_model.py
📍 Locatie: Regels 115-117 (in forward() methode)
❌ VOOR (Fout)
# Output layer x = self.layer_norm(x) logits = self.lm_head(x) return logits
✅ NA (Correct)
# Output layer with weight tying x = self.layer_norm(x) logits = x @ self.token_embeddings.weight.T \ + self.output_bias return logits

Waarom Dit Werkt?

x @ self.token_embeddings.weight.T vermenigvuldigt x (batch × seq × 512) met de getransponeerde embedding matrix (16.000 × 512).

Resultaat: batch × seq × 16.000 (logits voor elk token).

Daarna voegen we output_bias toe zodat elk token type een offset kan hebben.

Resultaat

Model doet hetzelfde, maar gebruikt geen aparte gewichtsmatrix.

🔧 Wijziging 3: Dynamische Causal Mask Generatie

📍 ARCHITECTURALE WIJZIGING: Flexible Attention Masking

Wat Was Het Probleem?

Original code: De causal mask werd eenmaal gemaakt in __init__ met vaste grootte 512×512.

Probleem 1: Als je tekstgeneratie wilt doen (Stap 5) met andere lengtes, kan dit niet.

Probleem 2: De mask was als "buffer" geregistreerd met persistent=False, dus moeilijk om op te slaan.

De Oplossing: Dynamische Generatie

Creëer de mask elke forward pass op basis van daadwerkelijke input lengte.

Wijziging 3A: __init__ aanpassing

📁 File: 03_build_transformer_model.py
📍 Locatie: Regels 74-75 (vervangen)
❌ VOOR
# Causal mask (ondersteuning voor # autoregressive generation) self.register_buffer( "causal_mask", self._create_causal_mask(max_seq_length), persistent=False )
✅ NA
# Max sequence length # (voor causal mask) self.max_seq_length = max_seq_length

Wijziging 3B: _create_causal_mask aanpassing

📁 File: 03_build_transformer_model.py
📍 Locatie: Regels 79-82
❌ VOOR
def _create_causal_mask(self, seq_length): """Creëer causal mask...""" mask = torch.triu( torch.ones(seq_length, seq_length) \ * float('-inf'), diagonal=1 ) return mask
✅ NA
def _create_causal_mask(self, seq_length, device): """Creëer causal mask...""" mask = torch.triu( torch.ones(seq_length, seq_length, device=device) * float('-inf'), diagonal=1 ) return mask

Wijziging 3C: Forward pass aanpassing

📁 File: 03_build_transformer_model.py
📍 Locatie: Regels 108-109
❌ VOOR
# Causal mask voor deze sequence length causal_mask = self.causal_mask[ :seq_length, :seq_length ].to(device)
✅ NA
# Causal mask dinamisch genereren # voor deze sequence length causal_mask = self._create_causal_mask( seq_length, device )

Waarom Dit Werkt?

Nu wordt de mask gegenereerd op basis van daadwerkelijke input grootte. Dit werkt voor alle lengtes (< 512).

Voor Stap 5 (tekstgeneratie): Kunnen we token-per-token genereren, het model zal de juiste mask creëren.

Resultaat

Flexibele causal masking die werkt voor willekeurige sequence lengths.
